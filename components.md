## 1.5 目录

### 目标

这是核心数据层，不是普通界面功能。必须先定义统一的目录 JSON 和目录 API，供 `1.1`、`1.6`、`1.7` 复用，而且目录数据结构必须兼容项目现有的 save/load 和 undo/redo 序列化方式。

目录的 JSON 设计必须遵循现有的序列化数据结构，不能脱离当前 document snapshot / node snapshot 格式单独设计。

做一个目录区域，并且建立一套目录数据结构。目录不是纯 UI，而是后续功能共用的数据层。

### 需要实现

- 有一个目录区域 / 目录面板
- 可把组件手动拖进目录
- 目录项可以反映结构顺序
- 目录项点击后可定位到对应节点 / slide
- 目录允许手动维护，不要求自动推导
- 必须定义一个目录 JSON 数据结构
- 必须提供统一的目录 API
- 目录 JSON 不能脱离现有文档结构单独设计
- 目录 JSON 必须能进入当前 `nodes[].data` 的序列化链路

### 目录 JSON

当前项目的 save/load 顶层结构是：

```json
{
  "schemaVersion": 1,
  "documentId": "doc-id",
  "revision": 0,
  "savedAt": "time",
  "meta": {},
  "view": {},
  "nodes": [],
  "drawings": []
}
```

当前项目的节点快照结构是：

```json
{
  "id": "node-id",
  "type": "component-type",
  "parentId": null,
  "x": 0,
  "y": 0,
  "rotation": 0,
  "scaleX": 1,
  "scaleY": 1,
  "visible": true,
  "opacity": 1,
  "focusPositionMode": null,
  "savedFocus": null,
  "data": {}
}
```

因此目录 JSON 不能直接另外发明成顶层文档结构，必须挂在一个专门的目录组件节点的 `data` 里，这样现有的 save/load 和 history 才能直接复用。

目录组件的 `data` 至少需要包含：

```json
{
  "items": [
    {
      "id": "catalog-item-1",
      "nodeId": "node-1",
      "title": "Chapter 1",
      "parentId": null,
      "order": 0,
      "collapsed": false
    }
  ]
}
```

### 目录 API

至少需要有这些能力：

- 创建目录项
- 删除目录项
- 更新目录项标题
- 调整顺序
- 移动父子关系
- 展开 / 折叠目录项
- 根据 `nodeId` 查目录项
- 根据目录顺序返回遍历结果

### 被依赖关系

- `1.1 Mind Map Branch` 要调用目录 API，同步树结构和折叠状态
- `1.6 Node 播放功能` 要基于目录 API 提供的顺序做播放
- `1.7 Node Treemap Chart 多级视图` 要基于目录 JSON 生成 treemap
- 后续如果做 outline，也应该优先复用目录组件里的目录 JSON，而不是再造一套树结构

### 完成标准

- 可以把节点拖入目录
- 目录顺序可调整
- 点击目录项可跳转到对应内容
- 目录 JSON 与当前 `nodes[].data` 序列化结构兼容
- 有统一可复用的目录 API
- save/load 可以恢复目录结构
- undo/redo 可以恢复目录结构变化

## 1.6 Node 播放功能

### 目标

必须基于 `1.5 目录` 的目录 API 和目录 JSON 实现，不能自己维护一套播放树结构。

基于目录顺序或树结构，实现节点播放能力。

### 需要实现

- 支持按结构顺序播放
- 支持键盘控制方向
- 例如：
- `1 -> 1.1 -> 1.1.1 -> 1.2 -> 2`
- 基于 1.5 的目录 API 和目录 JSON 推导播放顺序
- 左右键逻辑明确为：
- `←` 上一步
- `→` 下一步

### 完成标准

- 可进入播放模式
- 可按顺序切换节点
- 视口能跟随播放聚焦
